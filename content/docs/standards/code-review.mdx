# Code Review Guidelines

Code reviews are essential for maintaining code quality, sharing knowledge, and helping team members grow. This guide provides clear standards for both reviewers and authors.

## üéØ **Why Code Reviews Matter**

- **Quality Assurance**: Catch bugs before they reach production
- **Knowledge Sharing**: Learn from each other's approaches
- **Team Growth**: Help junior developers improve their skills
- **Consistency**: Maintain coding standards across the team
- **Documentation**: Ensure code is self-documenting and maintainable

## üìã **Review Checklist**

### **Before Starting Review**

- [ ] **Understand the context** - Read the PR description and related issues
- [ ] **Check the scope** - Ensure changes align with the task
- [ ] **Review the diff** - Look at the actual code changes
- [ ] **Run locally** (if needed) - Test the changes yourself

### **Code Quality Checks**

#### **Functionality**

- [ ] **Does it work?** - Code performs the intended function
- [ ] **Edge cases handled** - Proper error handling and validation
- [ ] **Performance** - No obvious performance issues
- [ ] **Security** - No security vulnerabilities

#### **Code Structure**

- [ ] **Single Responsibility** - Each function/component has one clear purpose
- [ ] **DRY Principle** - No unnecessary code duplication
- [ ] **Readable** - Code is self-documenting and easy to understand
- [ ] **Consistent** - Follows team coding standards

#### **Testing**

- [ ] **Tests included** - New features have appropriate tests
- [ ] **Tests pass** - All existing tests still pass
- [ ] **Test coverage** - Critical paths are tested
- [ ] **Test quality** - Tests are meaningful and maintainable

#### **Documentation**

- [ ] **Comments** - Complex logic is explained
- [ ] **README updates** - Documentation is updated if needed
- [ ] **Type definitions** - TypeScript types are clear and accurate
- [ ] **API documentation** - Public APIs are documented

## üö´ **Common Issues to Watch For**

### **Over-Engineering (Common with Junior Developers)**

<Callout type="warn">
  **Watch out for over-engineering!** Junior developers often create
  unnecessarily complex solutions.
</Callout>

#### **‚ùå Over-Engineered Examples**

```typescript
// ‚ùå Over-engineered: Complex abstraction for simple task
interface UserDataProcessor<T extends UserData> {
  process(data: T): ProcessedUserData<T>;
  validate(data: T): ValidationResult;
  transform(data: T): TransformedUserData<T>;
}

class UserDataProcessorImpl<T extends UserData>
  implements UserDataProcessor<T>
{
  constructor(
    private validator: Validator<T>,
    private transformer: Transformer<T>
  ) {}

  process(data: T): ProcessedUserData<T> {
    const validation = this.validate(data);
    if (!validation.isValid) throw new ValidationError(validation.errors);
    return this.transform(data);
  }
  // ... more complex code
}

// ‚úÖ Simple and clear
function processUserData(user: User): ProcessedUser {
  if (!user.email || !user.name) {
    throw new Error("Invalid user data");
  }

  return {
    id: user.id,
    email: user.email.toLowerCase(),
    name: user.name.trim(),
    createdAt: new Date(),
  };
}
```

#### **‚ùå Over-Engineered: Unnecessary Design Patterns**

```typescript
// ‚ùå Over-engineered: Factory pattern for simple object creation
interface ButtonFactory {
  createButton(type: "primary" | "secondary"): Button;
}

class ButtonFactoryImpl implements ButtonFactory {
  createButton(type: "primary" | "secondary"): Button {
    switch (type) {
      case "primary":
        return new PrimaryButton();
      case "secondary":
        return new SecondaryButton();
    }
  }
}

// ‚úÖ Simple and direct
function createButton(type: "primary" | "secondary"): Button {
  return type === "primary" ? new PrimaryButton() : new SecondaryButton();
}
```

### **Other Common Issues**

#### **‚ùå Poor Variable Names**

```typescript
// ‚ùå Unclear
const d = new Date();
const u = users.filter((x) => x.a > 18);

// ‚úÖ Clear
const currentDate = new Date();
const adultUsers = users.filter((user) => user.age > 18);
```

#### **‚ùå Magic Numbers**

```typescript
// ‚ùå Magic numbers
if (user.score > 85) {
  return "excellent";
}

// ‚úÖ Named constants
const EXCELLENT_SCORE_THRESHOLD = 85;
if (user.score > EXCELLENT_SCORE_THRESHOLD) {
  return "excellent";
}
```

#### **‚ùå Deep Nesting**

```typescript
// ‚ùå Hard to read
if (user) {
  if (user.profile) {
    if (user.profile.settings) {
      if (user.profile.settings.notifications) {
        return user.profile.settings.notifications.email;
      }
    }
  }
}

// ‚úÖ Early returns
if (!user?.profile?.settings?.notifications) {
  return null;
}
return user.profile.settings.notifications.email;
```

## üí¨ **How to Give Feedback**

### **Be Constructive, Not Critical**

#### **‚ùå Unhelpful Feedback**

- "This is wrong"
- "This doesn't make sense"
- "You should know this"

#### **‚úÖ Constructive Feedback**

- "This approach might cause issues because..."
- "Consider using X instead of Y for better performance"
- "Here's an alternative that might be clearer..."

### **Use the "Sandwich" Approach**

1. **Start positive**: "Great work on the feature implementation!"
2. **Provide feedback**: "I noticed a few areas we could improve..."
3. **End positive**: "Overall, this is a solid solution!"

### **Be Specific**

#### **‚ùå Vague**

- "This could be better"
- "Add more tests"

#### **‚úÖ Specific**

- "This function is doing too many things - consider breaking it into smaller functions"
- "Add a test case for when the user input is empty"

## üìù **PR Title Guidelines**

### **Format**
```
<type>: <brief description> (<issue number>)
```

### **Examples**
- `feat: add user authentication system (#123)`
- `fix: resolve login validation error (#456)`
- `docs: update API documentation (#789)`
- `refactor: improve component structure (#101)`
- `perf: optimize dashboard loading time (#234)`
- `test: add unit tests for user service (#567)`

### **Types**
- **`feat`** - New features or functionality
- **`fix`** - Bug fixes and issue resolutions
- **`docs`** - Documentation changes and updates
- **`style`** - Code style changes (formatting, whitespace, etc.)
- **`refactor`** - Code refactoring without functional changes
- **`test`** - Adding or updating tests
- **`chore`** - Maintenance tasks, dependencies, build changes
- **`perf`** - Performance improvements
- **`ci`** - Continuous integration changes
- **`build`** - Build system or external dependencies changes

### **Best Practices**
- **Keep titles under 50 characters** for better readability
- **Use present tense** ("add" not "added")
- **Be specific but concise** - describe what, not how
- **Include issue number** when applicable for traceability
- **Use lowercase** for consistency with commit messages
- **Avoid ending with periods** - titles are not sentences
- **Use imperative mood** - "Add feature" not "Adding feature"

### **Common Patterns**
```markdown
# Good Examples
feat: add dark mode toggle (#123)
fix: resolve memory leak in dashboard (#456)
docs: update setup instructions (#789)
refactor: extract user validation logic (#101)
perf: optimize image loading (#234)

# Avoid These
Added new feature (#123)                    # Wrong tense
FEAT: Add new feature (#123)                # Wrong case
feat: add new feature that does something (#123)  # Too long
feat: add new feature. (#123)               # Unnecessary period
```

## üéØ **Review Response Guidelines**

### **For Authors**

#### **When Receiving Feedback**

1. **Read carefully** - Understand what's being suggested
2. **Ask questions** - If feedback is unclear, ask for clarification
3. **Consider alternatives** - Think about different approaches
4. **Implement changes** - Make the requested improvements
5. **Respond professionally** - Thank reviewers for their time

#### **When Disagreeing**

- **Explain your reasoning** - Why you think your approach is better
- **Provide evidence** - Show examples or research
- **Be open to discussion** - Consider the reviewer's perspective
- **Escalate if needed** - Involve team lead for complex decisions

### **For Reviewers**

#### **When Approving**

- **Be confident** - Only approve if you're sure the code is ready
- **Leave positive feedback** - Acknowledge good work
- **Suggest improvements** - Even approved code can be improved

#### **When Requesting Changes**

- **Be specific** - Explain exactly what needs to change
- **Provide examples** - Show how to improve the code
- **Set expectations** - Let the author know what you're looking for
- **Follow up** - Check back on the changes

## üöÄ **Best Practices**

### **Review Size**

- **Keep PRs small** - Easier to review and understand
- **One feature per PR** - Avoid mixing unrelated changes
- **Break down large changes** - Split into multiple PRs if needed

### **Review Timing**

- **Review promptly** - Don't let PRs sit for days
- **Set expectations** - Let authors know when you'll review
- **Batch reviews** - Review multiple PRs at once if possible

### **Learning Opportunities**

- **Explain the "why"** - Help authors understand the reasoning
- **Share knowledge** - Point out useful patterns or techniques
- **Encourage questions** - Make it safe to ask for help

## üéì **For Junior Developers**

### **What to Focus On**

1. **Code readability** - Can others understand your code?
2. **Error handling** - What happens when things go wrong?
3. **Edge cases** - Have you considered all scenarios?
4. **Performance** - Is your code efficient?
5. **Testing** - Does your code work as expected?

### **Common Mistakes to Avoid**

- **Over-engineering** - Keep solutions simple
- **Copy-paste code** - Understand what you're copying
- **Ignoring feedback** - Learn from review comments
- **Rushing** - Take time to write good code
- **Not testing** - Always test your changes
