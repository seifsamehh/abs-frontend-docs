# Frequently Asked Questions

## Getting Started

### How do I set up my development environment?

1. Install Node.js 20+ (use nvm for version management)
2. Install pnpm: `npm install -g pnpm`
3. Clone the repository
4. Run `pnpm install`
5. Copy `.env.example` to `.env.local`
6. Run `pnpm dev`

See [Getting Started](/docs/getting-started) for detailed instructions.

### What IDE should I use?

We recommend **VS Code** with these extensions:

- ESLint
- Prettier
- Tailwind CSS IntelliSense
- TypeScript + JavaScript Language Features
- GitLens

### How do I get access to the repositories?

Contact the Frontend Lead or your Team Lead to be added to the GitHub organization.

## Development

### What's the difference between `npm`, `yarn`, and `pnpm`?

We use **pnpm** for all projects. It's faster and uses less disk space through hard links. Always use `pnpm` commands:

```bash
pnpm install    # Install dependencies
pnpm add <pkg>  # Add a package
pnpm dev        # Run development server
```

### When should I use Server Components vs Client Components?

<Tabs items={['Server Components', 'Client Components']}>
  <Tab value="Server Components">
    **Use Server Components (default) when:**
    - Fetching data
    - Accessing backend resources directly
    - Keeping sensitive information on the server
    - Reducing client-side JavaScript

    ```tsx
    // No 'use client' directive = Server Component
    async function UserList() {
      const users = await db.users.findMany();
      return <ul>{users.map(u => <li key={u.id}>{u.name}</li>)}</ul>;
    }
    ```

</Tab>
  <Tab value="Client Components">
    **Use Client Components when:**
    - Using hooks (useState, useEffect, etc.)
    - Adding event listeners
    - Using browser-only APIs
    - Using React Context

    ```tsx
    'use client';

    function Counter() {
      const [count, setCount] = useState(0);
      return <button onClick={() => setCount(c => c + 1)}>{count}</button>;
    }
    ```

</Tab>
</Tabs>

### How do I handle environment variables?

- **Server-side only:** Use `process.env.VARIABLE_NAME`
- **Client-side:** Prefix with `NEXT_PUBLIC_` like `NEXT_PUBLIC_API_URL`
- **Never expose secrets** to the client

```typescript
// Server-side only
const secret = process.env.DATABASE_URL;

// Available on client
const apiUrl = process.env.NEXT_PUBLIC_API_URL;
```

### Where should I put shared components?

```
src/
├── components/
│   ├── ui/          # Generic UI components (Button, Input, etc.)
│   ├── features/    # Feature-specific components
│   └── layout/      # Layout components (Header, Footer, etc.)
```

## Testing

### How do I run tests?

```bash
pnpm test           # Run all tests
pnpm test:watch     # Run in watch mode
pnpm test:coverage  # Run with coverage report
pnpm test:e2e       # Run E2E tests (Playwright)
```

### What should I test?

**Priority order:**

1. **Business logic** - Calculations, transformations, validations
2. **User interactions** - Forms, buttons, navigation
3. **Edge cases** - Error states, empty states, loading states
4. **Integration points** - API calls, state management

### Do I need to write tests for everything?

Focus on testing behavior that matters to users and business logic. Don't test implementation details or third-party library internals.

## Git & Workflow

### What's the commit message format?

Use [Conventional Commits](https://www.conventionalcommits.org/):

```bash
feat: add user authentication
fix: resolve login redirect issue
docs: update API documentation
style: format code with prettier
refactor: extract validation logic
test: add unit tests for auth
chore: update dependencies
```

### How do I handle merge conflicts?

```bash
# Fetch latest changes
git fetch origin

# Rebase your branch
git rebase origin/main

# Resolve conflicts in your editor, then:
git add .
git rebase --continue
```

### Should I squash my commits?

Yes, squash related commits before merging:

```bash
# Interactive rebase to squash last 3 commits
git rebase -i HEAD~3
```

## Deployment

### How do I deploy to staging?

Push to the `develop` branch. CI/CD will automatically deploy to staging.

### How do I deploy to production?

1. Create a PR from `develop` to `main`
2. Get code review approval
3. Merge the PR
4. CI/CD will automatically deploy to production

### How do I roll back a deployment?

Contact the Frontend Lead or use Vercel dashboard to roll back to the previous deployment.

## Troubleshooting

### My changes aren't showing up

```bash
# Clear Next.js cache
rm -rf .next

# Restart dev server
pnpm dev
```

### TypeScript errors but code runs fine

```bash
# Restart TypeScript server in VS Code
Cmd/Ctrl + Shift + P → "TypeScript: Restart TS Server"
```

### "Module not found" errors

```bash
# Reinstall dependencies
rm -rf node_modules pnpm-lock.yaml
pnpm install
```
