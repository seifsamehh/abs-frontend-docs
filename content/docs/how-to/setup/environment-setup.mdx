## Making Your First Contribution

### 1. Get Assigned to a Task

As a new team member, you'll typically start with:

- **Small bug fixes** - Get familiar with the codebase
- **Documentation updates** - Learn the project structure
- **UI improvements** - Work on existing components
- **Feature additions** - Build on existing functionality

Your team lead or mentor will assign you appropriate tasks based on your experience level.

### 2. Understand the Task

Before starting work:

- **Read the task description** thoroughly
- **Ask questions** if anything is unclear
- **Review related code** to understand the context
- **Check existing tests** to understand expected behavior
- **Discuss approach** with your mentor or team lead

### 3. Set Up Your Development Environment

<Callout type="warn">
  **Daily Workflow Tip:** At the start of each day before you begin working,
  always pull the latest changes from the `stage` branch to ensure your local
  environment is up to date with the team's latest features.
</Callout>

#### Our Branching Strategy

| Branch      | Owner           | Purpose                                                        |
| ----------- | --------------- | -------------------------------------------------------------- |
| `main`      | Frontend Leader | Production-ready, stable code only                             |
| `stage`     | Coordinator     | Integration branch where features are tested before production |
| `feature/*` | Developers      | Individual feature development                                 |

**Workflow:** `feature/*` ‚Üí `stage` ‚Üí `main`

All feature branches should be created from and merged into `stage` first. The coordinator reviews and tests features in `stage`, then the frontend leader promotes stable code to `main`.

```bash
# Start of day: Pull latest from stage to get team's latest work
git checkout stage
git pull origin stage

# Create your feature branch FROM stage (not main)
git checkout -b feature/your-task-name

# Install dependencies (if needed)
npm install

# Start development server
npm run dev
```

### 4. Make Your Changes

Follow our development standards:

```typescript
// Example: Adding a new component
import { FC } from "react";
import { cn } from "@/lib/utils";

interface ButtonProps {
  children: React.ReactNode;
  variant?: "primary" | "secondary";
  size?: "sm" | "md" | "lg";
  disabled?: boolean;
  onClick?: () => void;
}

export const Button: FC<ButtonProps> = ({
  children,
  variant = "primary",
  size = "md",
  disabled = false,
  onClick,
}) => {
  return (
    <button
      className={cn("rounded-md font-medium transition-colors", {
        "bg-blue-600 text-white hover:bg-blue-700": variant === "primary",
        "bg-gray-200 text-gray-900 hover:bg-gray-300": variant === "secondary",
        "px-2 py-1 text-sm": size === "sm",
        "px-4 py-2 text-base": size === "md",
        "px-6 py-3 text-lg": size === "lg",
        "opacity-50 cursor-not-allowed": disabled,
      })}
      disabled={disabled}
      onClick={onClick}
    >
      {children}
    </button>
  );
};
```

### 5. Test Your Changes

```bash
# Run the development server
npm run dev

# Check for TypeScript errors
npm run type-check

# Run linting
npm run lint

# Run tests
npm run test

# Format code
npm run format
```

### 6. Commit Your Changes

```bash
# Stage your changes
git add .

# Commit with conventional commit format
git commit -m "feat: add new button component with variants"

# Push to your branch
git push origin feature/your-task-name
```

### 7. Create Pull Request

1. Go to the repository on GitHub
2. Click "New Pull Request"
3. Select your branch
4. **Follow PR Guidelines:**
   - Use proper [PR Title Format](/docs/getting-started/first-contribution#pr-title): `feat: add new feature (#123)`
   - Fill out the [PR Description Template](/docs/getting-started/first-contribution#pr-description-template)
   - Include **Screenshots** if UI changes
   - Add **Testing notes** if applicable
   - Include **Any questions** for reviewers
5. Request review from your mentor or team lead

### 8. Address Feedback

- **Respond to comments** promptly
- **Make requested changes** carefully
- **Ask for clarification** if needed
- **Test changes** after each update
- **Update your PR** with new commits

### 9. Celebrate Your First Contribution! üéâ

Once your PR is approved and merged:

- **Update your local main branch**
- **Delete your feature branch**
- **Document any learnings** for future reference
- **Ask for your next task** from your team lead

## Development Workflow

### Daily Workflow

1. **Start of Day**

   ```bash
   # Pull latest from stage to get team's latest work
   git checkout stage
   git pull origin stage

   # Create your feature branch FROM stage (not main)
   git checkout -b feature/your-task-name

   # Install dependencies (if needed)
   pnpm install

   # Start development server
   pnpm dev
   ```

2. **Working on Features**

   ```bash
   # Make changes and test
   # Commit regularly with good messages
   git add .
   git commit -m "feat: implement new feature"

   # Keep your branch updated with stage
   git fetch origin stage
   git rebase origin/stage
   ```

3. **End of Day**

   ```bash
   # Push your work
   git push origin feature/your-task-name

   # Create PR targeting stage branch (not main)
   ```

## Common Tasks

### üÜï Implementing New Features

1. **Plan & Design**

   - Create feature branch from `stage`
   - Write technical specification
   - Design component architecture
   - Identify dependencies and APIs

2. **Development**

   - Write unit tests first (TDD approach)
   - Implement feature incrementally
   - Follow coding standards and patterns
   - Add proper TypeScript types

3. **Testing & Review**

   - Run all tests locally
   - Test in different browsers/devices
   - Create pull request with detailed description
   - Address code review feedback

4. **Deployment**
   - Merge to `stage` after approval
   - Monitor deployment and user feedback
   - Update documentation

### üîß Refactoring Code

1. **Assessment**

   - Identify code smells and technical debt
   - Analyze impact and dependencies
   - Plan refactoring strategy

2. **Implementation**

   - Create feature branch for refactoring
   - Make small, incremental changes
   - Maintain existing functionality
   - Update tests as needed

3. **Validation**
   - Run comprehensive test suite
   - Verify no regression in functionality
   - Performance testing if applicable
   - Code review and approval

### üìù Adding Documentation

1. **Content Creation**

   - Create new `.mdx` files in appropriate sections
   - Follow documentation standards and templates
   - Include code examples and screenshots
   - Add proper frontmatter metadata

2. **Integration**
   - Update navigation in `meta.json` files
   - Add cross-references and links
   - Test documentation locally
   - Review with team members

### ‚úèÔ∏è Editing & Updating

1. **Content Updates**

   - Identify outdated information
   - Update code examples and screenshots
   - Revise procedures and workflows
   - Check for broken links

2. **Version Control**
   - Create branch for documentation updates
   - Make atomic commits for each change
   - Test changes locally
   - Submit pull request with clear description

### üêõ Fixing Bugs

1. **Investigation**

   - Reproduce the issue locally
   - Identify root cause and scope
   - Check existing issues and discussions
   - Create test case (if missing)

2. **Resolution**

   - Implement fix with minimal changes
   - Add or update tests
   - Verify fix resolves the issue
   - Update documentation if needed

3. **Deployment**
   - Test fix in staging environment
   - Create pull request with detailed description
   - Monitor fix in production
   - Close related issues

### üé® UI/UX Improvements

1. **Design Review**

   - Analyze current user experience
   - Identify improvement opportunities
   - Create mockups or wireframes
   - Get design approval

2. **Implementation**

   - Update components and styles
   - Ensure responsive design
   - Test accessibility compliance
   - Validate across browsers

3. **Testing**
   - User acceptance testing
   - Performance impact assessment
   - Cross-browser compatibility
   - Mobile device testing

### ‚ö° Performance Optimization

1. **Analysis**

   - Profile application performance
   - Identify bottlenecks and slow queries
   - Measure Core Web Vitals
   - Set performance budgets

2. **Optimization**

   - Implement code splitting
   - Optimize images and assets
   - Improve database queries
   - Add caching strategies

3. **Validation**
   - Measure performance improvements
   - Run Lighthouse audits
   - Test on slow connections
   - Monitor production metrics

### üîí Security Updates

1. **Assessment**

   - Review security advisories
   - Audit dependencies for vulnerabilities
   - Check for security best practices
   - Identify potential attack vectors

2. **Implementation**

   - Update vulnerable dependencies
   - Implement security headers
   - Add input validation and sanitization
   - Review authentication and authorization

3. **Testing**
   - Run security scans
   - Penetration testing if needed
   - Review with security team
   - Monitor for security issues

### üß™ Testing Tasks

1. **Test Development**

   - Write unit tests for new features
   - Create integration tests
   - Add end-to-end test scenarios
   - Update test documentation

2. **Test Maintenance**

   - Fix failing tests
   - Update outdated test data
   - Improve test coverage
   - Optimize test performance

3. **Test Execution**
   - Run test suites locally
   - Debug test failures
