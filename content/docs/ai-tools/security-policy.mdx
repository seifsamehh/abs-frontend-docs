This document outlines the security guidelines and policies for using AI tools in our frontend development workflow.

## üö® Critical Security Rules

### 1. Never Include Sensitive Data

**‚ùå NEVER include in AI prompts:**

- API keys or secrets
- Passwords or authentication tokens
- User data or personal information
- Database credentials
- Internal business logic
- Proprietary algorithms

**‚úÖ Use instead:**

- Placeholder data (`"user123"`, `"example@email.com"`)
- Generic examples
- Mock data
- Configuration placeholders

### 2. Code Sanitization

**Before using AI-generated code:**

- [ ] Remove any hardcoded credentials
- [ ] Replace placeholder data with proper variables
- [ ] Validate all external inputs
- [ ] Review for security vulnerabilities
- [ ] Test with security tools

### 3. Input Validation

**All AI-generated code must include:**

- Zod schemas for all external inputs
- Proper type checking
- Input sanitization
- Error handling for invalid inputs

## üîç Common Security Issues

### 1. XSS (Cross-Site Scripting)

**Problem:** Unsanitized user input

```typescript
// ‚ùå Dangerous
<div dangerouslySetInnerHTML={{ __html: userInput }} />

// ‚úÖ Safe
<div>{sanitizeHtml(userInput)}</div>
```

### 2. Injection Attacks

**Problem:** Unvalidated external inputs

```typescript
// ‚ùå Dangerous
const query = `SELECT * FROM users WHERE id = ${userId}`;

// ‚úÖ Safe
const query = "SELECT * FROM users WHERE id = ?";
const result = await db.query(query, [userId]);
```

### 3. Insecure Authentication

**Problem:** Client-side only auth checks

```typescript
// ‚ùå Dangerous
if (localStorage.getItem("token")) {
  // Allow access
}

// ‚úÖ Safe
// Server-side validation required
const isValid = await validateToken(token);
```

### 4. Exposed Secrets

**Problem:** Hardcoded credentials

```typescript
// ‚ùå Dangerous
const API_KEY = "sk-1234567890abcdef";

// ‚úÖ Safe
const API_KEY = process.env.NEXT_PUBLIC_API_KEY;
```

## üîê Secure AI Usage Patterns

### 1. Environment Variables

```typescript
// Use environment variables for sensitive data
const config = {
  apiUrl: process.env.NEXT_PUBLIC_API_URL,
  apiKey: process.env.API_KEY, // Server-side only
};
```

### 2. Input Validation

```typescript
import { z } from "zod";

const UserSchema = z.object({
  name: z.string().min(1).max(100),
  email: z.string().email(),
  age: z.number().min(0).max(120),
});

// Validate all inputs
const validatedData = UserSchema.parse(userInput);
```

### 3. Secure API Calls

```typescript
// Use secure HTTP methods and headers
const response = await fetch("/api/users", {
  method: "POST",
  headers: {
    "Content-Type": "application/json",
    Authorization: `Bearer ${token}`,
  },
  body: JSON.stringify(validatedData),
});
```

### 4. Error Handling

```typescript
try {
  const result = await riskyOperation();
  return result;
} catch (error) {
  // Log error securely (no sensitive data)
  console.error("Operation failed:", error.message);
  // Return generic error to user
  throw new Error("Operation failed");
}
```

## üö´ Prohibited Practices

### 1. Data Exposure

- **Never log sensitive data** - No passwords, tokens, or user data in logs
- **Never expose internal APIs** - Don't expose internal endpoints
- **Never hardcode secrets** - Use environment variables

### 2. Insecure Patterns

- **No client-side auth** - Always validate server-side
- **No SQL injection** - Use parameterized queries
- **No XSS vulnerabilities** - Sanitize all user input

### 3. Poor Error Handling

- **No detailed error messages** - Don't expose system internals
- **No stack traces** - Don't expose code structure
- **No debug information** - Remove debug info in production
